## 1. Overview

**Wave** is an accounting + invoicing SaaS platform. Unlike most providers in
Shielva, Wave exposes a single **GraphQL endpoint** rather than a REST surface:

```
POST https://gql.waveapps.com/graphql/public
Authorization: Bearer <full_access_token>
Content-Type:  application/json
{ "query": "...", "variables": {...} }
```

This connector — `WaveConnector` (`CONNECTOR_TYPE = "wave"`,
`AUTH_TYPE = "api_key"`) — wraps the operational surfaces a Shielva tenant
typically needs from a Wave business:

| Surface          | GraphQL ops                                              | Capability                            |
|------------------|----------------------------------------------------------|---------------------------------------|
| Identity         | `user`                                                   | Authenticated user node               |
| Businesses       | `businesses`, `business(id)`                             | List + read businesses on the account |
| Customers        | `business.customers`, `customerCreate`, `customerPatch`  | List + create customers               |
| Invoices         | `business.invoices`, `invoiceCreate`                     | List + create invoices                |
| Products         | `business.products`, `productCreate`                     | List + create products                |
| Accounts         | `business.accounts`                                      | Read chart of accounts                |
| Transactions     | `business.transactions`                                  | List ledger transactions              |
| Sales taxes      | `business.salesTaxes`                                    | List sales tax rates                  |
| Users            | `user`                                                   | Identity probe                        |

The connector normalises **customers** and **invoices** into `NormalizedDocument`
(id = `f"{tenant_id}_{source_id}"`), surfaces standalone `async def` methods per
user-requested operation (OCP), retries 429/5xx with exponential backoff, and
never embeds raw HTTP or query strings in `connector.py` (SOC — all HTTP lives
in `client/http_client.py::WaveHTTPClient` and every operation string lives in
`helpers/queries.py`).

The connector deliberately does **not** depend on a GraphQL client library
(`gql`, `graphql-core`). The Wave surface area is small (~12 ops) and the
queries are stored as static strings — a thin `httpx.AsyncClient` wrapping
`POST { query, variables }` is both simpler and one fewer transitive surface to
audit.

## 3. Auth Flow

Wave's modern public API uses **personal access token authentication** for
server-to-server integrations. A "full access token" is generated by the
business owner from `https://accounts.waveapps.com/applications/` and copied
once into the ACP install form.

### Credentials
- `access_token` — Wave full-access personal access token. install_field
  (type `secret`, required). Stored encrypted at rest via the credential vault.
- `business_id` — Optional default Business ID (UUID). install_field (type
  `string`, optional). When set, the per-call `business_id` argument can be
  omitted and falls back to this value.

### Header contract
Every request to `https://gql.waveapps.com/graphql/public`:

```
Authorization: Bearer <access_token>
Content-Type:  application/json
Accept:        application/json
```

### Lifecycle
- `install()` validates `access_token` is non-empty. Does **not** call the API.
- `authorize(...)` — surface-compatible no-op (`api_key` has no exchange);
  returns a `TokenInfo` whose `access_token` is the configured token.
- `health_check()` — `POST` a minimal `{ user { id } }` GraphQL query.
- `ensure_token()` — N/A (no token lifecycle, no refresh, no expiry).

## 5. Key API Endpoints & Methods

Every method below is a standalone public `async def` on `WaveConnector`. SOC:
each method delegates to `WaveHTTPClient.execute(query, variables, ...)`; the
query strings live in `helpers/queries.py`.

| Method                                  | GraphQL op             | Notes                                          |
|-----------------------------------------|------------------------|------------------------------------------------|
| `install()`                             | (lifecycle)            | Validate `access_token`; initialise client.    |
| `health_check()`                        | `user`                 | Lightweight `{ user { id } }` probe.           |
| `sync(since, full, kb_id, webhook_url)` | iterates customers + invoices | Calls `ingest_document` per item.       |
| `get_user()`                            | `user`                 | Authenticated user node.                       |
| `list_businesses(page, page_size)`      | `businesses`           | `(page, pageSize)` paging.                     |
| `get_business(business_id)`             | `business(id)`         |                                                |
| `list_customers(business_id, ...)`      | `business.customers`   |                                                |
| `get_customer(business_id, customer_id)`| `business.customer`    |                                                |
| `create_customer(business_id, name, email)` | `customerCreate`   | Input: `CustomerCreateInput`.                  |
| `list_invoices(business_id, ..., status)`   | `business.invoices`| Optional `status` filter.                      |
| `get_invoice(business_id, invoice_id)`  | `business.invoice`     |                                                |
| `create_invoice(business_id, customer_id, items, due_date)` | `invoiceCreate` | Items: `[{product_id, quantity?, unit_price?, description?}]`. |
| `list_products(business_id, ...)`       | `business.products`    |                                                |
| `create_product(business_id, name, unit_price, ...)` | `productCreate` |                                            |
| `list_accounts(business_id)`            | `business.accounts`    | Chart of accounts.                             |
| `list_transactions(business_id, ...)`   | `business.transactions`| Optional `from`/`to` date range.               |
| `list_sales_taxes(business_id)`         | `business.salesTaxes`  | Sales-tax rates.                               |
| `list_users()`                          | `user`                 | Alias for `get_user`, returns single-element list shape. |

### GraphQL request envelope

Wave's GraphQL POST body for every operation:

```json
{
  "query": "query ListBusinesses($page: Int!, $pageSize: Int!) { ... }",
  "variables": { "page": 1, "pageSize": 50 }
}
```

Response on success:

```json
{ "data": { "businesses": { ... } } }
```

Response on partial / total failure (HTTP 200 is still possible):

```json
{ "data": null, "errors": [ { "message": "...", "extensions": { "code": "..." } } ] }
```

The `WaveHTTPClient` always inspects `errors` even on HTTP 200 and surfaces them
as `WaveError` carrying the joined messages (this is the GraphQL spec contract).

## 6. Error Handling

| HTTP / GraphQL                     | Wave meaning                          | Mapped to                                                       |
|------------------------------------|---------------------------------------|-----------------------------------------------------------------|
| 400                                | Bad request / malformed query         | `WaveValidationError`                                           |
| 401                                | Access token invalid / revoked        | `WaveAuthError` → `AuthStatus.TOKEN_EXPIRED` + `OFFLINE`        |
| 403                                | Token lacks the required scope        | `WaveAuthError` → `AuthStatus.INVALID_CREDENTIALS` + `UNHEALTHY`|
| 404                                | Resource not found                    | `WaveNotFoundError`                                             |
| 429                                | Rate limit exceeded                   | `WaveRateLimitError` → `DEGRADED` + `CONNECTED`                 |
| 5xx                                | Provider outage                       | `WaveNetworkError` (retry with backoff)                         |
| 200 + `errors[]`                   | GraphQL validation / authorization    | `WaveError` carrying joined `errors[*].message`                 |
| `httpx.TimeoutException` / network | Transport flake                       | `WaveNetworkError` (retry with backoff)                         |

All in `exceptions.py` extending `WaveError`. Retry in
`client/http_client.py::_post` honours `_MAX_RETRIES = 3` with exponential
backoff `_BACKOFF_BASE * (2 ** attempt)` for 429 + 5xx + transport errors.
